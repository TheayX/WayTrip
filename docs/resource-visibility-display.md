# 资源可见性与展示语义约定

这份文档只收口项目里最常用、最稳定的判断规则，用来回答两个问题：

1. 一条记录还要不要显示
2. 如果显示，应该显示什么

详细按表展开版仍保留在 [_temp/resource-visibility-and-display-contract-by-table.md](../_temp/resource-visibility-and-display-contract-by-table.md) 里；日常开发默认先看这份短约定。

## 1. 总原则

所有资源展示统一分两层判断：

1. 先判断 `是否显示`
2. 再判断 `显示什么`

额外统一约束：

1. `--` 只用于普通字段空值，不用于主体失效
2. 用户端不暴露后台治理术语
3. 管理端优先保留排查问题所需的信息
4. 页面层不要散落硬编码，统一走共享常量和展示工具

## 2. 是否显示

### 2.1 主体主表

当前项目为了保持主列表整洁，主体主表默认继续过滤软删数据：

1. `spot` 管理主列表默认只显示 `is_deleted = 0`
2. `guide` 管理主列表默认只显示 `is_deleted = 0`
3. 用户端当前态列表、搜索、推荐、详情入口只显示当前可用主体

这意味着：

1. 主体被软删后，通常不再继续留在主列表里
2. 当前阶段重点不是“已删除主体是否继续挂在主表中”
3. 当前阶段重点是“它被别的历史数据引用时怎么显示”

### 2.2 强历史数据

下面这些链路在主体失效后仍然保留：

1. `order`
2. `user_spot_review`
3. `user_spot_view`
4. 管理端行为分析、审计、创建者、操作人等历史链路

规则是：

1. 不因为关联主体软删或硬删，就把整条历史记录过滤掉
2. 历史记录保留后，再对主体名称做语义降级
3. 用户端历史页如果主体失效，可以继续显示文字，但应禁用无效跳转

### 2.3 弱状态数据

下面这些链路在主体失效后应收口，不继续作为当前态展示：

1. `user_preference`
2. `user_spot_favorite`
3. `spot_banner`
4. `guide_spot_relation`
5. 各类前端详情缓存、预览缓存

规则是：

1. 用户端当前态只保留仍然有效的数据
2. 管理端如果需要治理，可保留必要的失效语义
3. 不为了“兼容”保留一套旧逻辑继续混用

## 3. 显示什么

### 3.1 用户端

用户端统一弱化：

1. 失效用户显示 `未知用户`
2. 失效管理员显示 `未知管理员`
3. 失效景点显示 `未知景点`
4. 失效攻略显示 `未知攻略`
5. 无法识别来源显示 `未知来源`

用户端禁止直接展示：

1. `已下架景点`
2. `已删除景点`
3. `已清除景点`
4. `已删除攻略`
5. `已停用管理员`
6. `已注销用户`

### 3.2 管理端

管理端保留更细粒度治理语义，但不做过度收紧：

1. 用户：`真实昵称 / 已注销用户 / 已清除用户 / 未知用户`
2. 管理员：`真实姓名或用户名 / 已停用管理员 / 已清除管理员 / 未知管理员`
3. 景点：`景点名 / 已下架景点 / 已删除景点 / 已清除景点 / 未知景点`
4. 攻略：`攻略标题 / 已删除攻略 / 已清除攻略 / 未知攻略`
5. 来源：`未知来源`

同时遵守一条实际口径：

1. 管理端能保留原始识别信息时，优先保留，方便定位问题
2. 只有继续直显会误导时，才切到治理语义

## 4. 页面层规则

### 4.1 用户端当前态页面

包括首页、发现、搜索、推荐、景点列表、攻略列表、下单入口。

统一规则：

1. 只展示当前可用主体
2. 命中失效缓存时，改成中性失效态
3. 不暴露后台治理术语

### 4.2 用户端历史页面

包括我的订单、我的评价、我的足迹、旧详情入口。

统一规则：

1. 历史记录可以保留
2. 主体失效后统一弱化成 `未知*`
3. 失效主体的跳转入口要禁用或拦截

### 4.3 管理端治理页面

包括用户管理、订单中心、评价管理、浏览行为、收藏分析、偏好分析、轮播管理、攻略管理、景点管理。

统一规则：

1. 尽量保留历史链路和治理链路
2. 主体主列表仍以整洁优先，默认不过度保留软删主体
3. 关联主体失效时必须有明确语义
4. 对已经失效且不可继续编辑的主体，禁用无效跳转

## 5. 查询与实现约束

### 5.1 查询层

1. 当前态查询优先过滤失效主体
2. 历史查询优先保留业务记录，再降级展示关联主体
3. 不允许把“主体失效”错误处理成“整条记录不显示”

### 5.2 组装层

1. 失效主体文案统一走共享常量
2. 前后端都要走统一展示工具
3. 页面层不要重复写一套 `未知景点`、`已删除景点` 之类 hardcode

### 5.3 交互层

1. 用户端命中失效主体时，优先禁跳转
2. 详情页命中失效资源时，统一进入中性失效态
3. 缓存回写前先校验缓存结构，不合格直接清理

## 6. 当前项目的落地口径

这轮开发以这几条为准：

1. `spot`、`guide` 管理主列表继续过滤软删主体
2. 重点治理订单、评价、浏览、轮播、攻略关联、创建者等引用链路
3. 用户端统一 `未知*`
4. 管理端保留治理语义，但允许保留识别信息帮助排查

后续如果要改“主表是否继续展示软删主体”，应单独作为产品和治理策略调整，不在页面里临时各改各的。
