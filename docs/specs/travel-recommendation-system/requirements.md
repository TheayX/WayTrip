# 需求文档

## 文档说明

- 文档范围：WayTrip 个性化旅游推荐系统
- 对齐基线：当前仓库实现
- 更新时间：2026-04-08
- 说明：本版同步四端重构后的功能边界，重点补齐 Web 端与小程序端一致性、发现页/附近/互动页等最新范围

## 产品简介

WayTrip 是一个面向旅游场景的个性化旅游推荐系统，包含：

- 小程序端：Uni-app + Vue 3 + Pinia
- Web 用户端：Vue 3 + Element Plus + Pinia
- 管理端：Vue 3 + Element Plus + ECharts
- 后端：Spring Boot 3 + MyBatis-Plus + MySQL + Redis

系统支持景点浏览、攻略阅读、评分评论、收藏、订单、推荐、附近探索、个人中心，以及管理端的内容维护和推荐调试。

## 术语

- 小程序端：面向移动端用户的 Uni-app 应用
- Web 用户端：面向 PC 用户的 Vue Web 应用
- 管理端：面向管理员的后台系统
- 推荐引擎：基于 ItemCF 的景点推荐逻辑
- 冷启动：用户交互不足时返回热门内容并引导设置偏好
- 逻辑删除：通过 `is_deleted=1` 屏蔽业务数据，不做物理删除
- 浏览行为：来自 `user_spot_view` 的景点访问记录，包含来源与停留时长
- 双端一致性：小程序端与 Web 用户端在核心业务能力、接口覆盖和关键用户链路上保持一致

- 攻略：列表、详情、分类、关联景点
- 评分评论：提交评分评论、读取用户评分、评论列表、景点评分聚合

## 功能需求

### 需求 1：小程序用户认证

用户可以通过微信登录进入系统；新用户先拿到 `openid`，绑定手机号后完成注册；老用户直接返回登录态。

验收标准：

1. 小程序登录调用微信登录能力并换取 OpenID。
2. OpenID 未注册时，仅返回绑定前上下文，不直接创建完整账户。
3. OpenID 已存在时，返回 token 和用户信息。
4. 绑定手机号后，系统创建用户或将 OpenID 合并到既有账户。
5. 登录后个人中心可读取昵称、头像和偏好信息。

### 需求 2：首页与轮播图

首页需要提供轮播图、热门推荐、个性化推荐、附近探索入口和冷启动引导。

验收标准：

1. 首页展示已启用轮播图。
2. 点击轮播图跳转至对应景点详情。
3. 首页展示热门景点列表。
4. 登录用户可查看个性化推荐并支持刷新。
5. 冷启动用户可看到偏好引导。
6. 首页展示附近探索入口，并可进入附近景点页。

### 需求 3：个性化推荐

系统根据评分、收藏、订单和浏览行为生成个性化推荐；当用户交互不足时使用冷启动策略。

验收标准：

1. 用户交互数达到阈值时返回个性化推荐。
2. 交互不足时返回热门推荐，并提示设置偏好。
3. 推荐结果过滤已评分、已收藏、已下单但未取消的景点。
4. 被取消订单的景点允许重新进入推荐结果。
5. 推荐支持刷新。
6. 浏览行为需要考虑来源与停留时长差异。
7. 推荐结果支持热度重排。
8. 管理端可查看推荐配置、状态与调试预览。

### 需求 4：景点浏览与搜索

用户可浏览、筛选、搜索景点，并查看完整详情。

验收标准：

1. 支持按地区、分类筛选。
2. 支持按热度、评分、价格排序。
3. 支持分页加载。
4. 支持关键字模糊搜索。
5. 景点详情展示图片、简介、票价、开放时间、地址、经纬度、评论、收藏状态、相似景点。
6. 小程序支持调用地图导航。
7. Web 支持基于浏览器定位查看附近景点。
8. 浏览行为可写入浏览记录，用于推荐与热度计算。

### 需求 5：发现页聚合浏览

用户需要从统一入口浏览推荐、景点与攻略内容，而不是在首页只看单一列表。

验收标准：

1. 提供发现页统一入口。
2. 发现页支持切换景点与攻略视图。
3. 可带预设筛选条件进入发现页。
4. 从发现页进入详情后返回，筛选状态需要尽量保持。

### 需求 6：旅游攻略

用户可浏览和阅读攻略，查看关联景点。

验收标准：

1. 支持攻略分页列表。
2. 支持按分类筛选。
3. 详情页支持富文本渲染。
4. 展示关联景点并支持跳转。
5. 已下架内容不在用户端展示。

### 需求 7：评分与评论

用户可对景点提交评分和评论。

验收标准：

1. 支持 1-5 分评分。
2. 再次评分时更新原记录。
3. 评论写入后可在详情页读取。
4. 评分写入后同步更新景点平均分和评分数。
5. 已完成订单可从订单链路进入评价区。

### 需求 8：收藏与互动管理

用户可收藏和取消收藏景点，并在统一互动页查看浏览、收藏与评价记录。

验收标准：

1. 支持收藏添加。
2. 支持取消收藏。
3. 支持收藏列表查询。
4. 收藏状态变更后前端立即可见。
5. 提供“我的互动”聚合页展示浏览、收藏、评价。

### 需求 9：订单创建与查询

用户可创建门票订单并查看列表与详情。

验收标准：

1. 创建订单时填写游玩日期、数量、联系人信息。
2. 新建订单默认状态为待支付。
3. 列表支持按状态筛选与分页。
4. 详情展示景点、数量、金额、状态与联系人信息。
5. 用户端可从个人中心直接跳转指定订单状态。

### 需求 10：模拟支付

订单支持模拟支付流程。

验收标准：

1. 待支付订单可发起支付。
2. 支付成功后状态更新为已支付。
3. 多次触发支付时不产生重复结果。

### 需求 11：用户取消订单

用户可取消待支付或已支付订单。

验收标准：

1. 允许取消待支付和已支付订单。
2. 已完成订单不可取消。
3. 重复取消已取消订单时按幂等处理。

### 需求 12：管理端订单流转

管理员可查看订单并对订单执行完成、退款、取消、恢复操作。

验收标准：

1. 已支付订单可标记完成。
2. 已支付订单可执行退款。
3. 待支付订单可由管理员取消。
4. 已完成订单可恢复为已支付。

### 需求 13：个人中心与设置

用户可在个人中心查看与管理账户信息，并进入设置与互动相关页面。

验收标准：

1. 展示昵称、头像、偏好标签。
2. 提供订单入口。
3. 提供互动入口。
4. 提供设置入口。
5. 支持头像上传、修改密码、注销账户。
6. 浏览足迹统计可在个人中心读取。

### 需求 14：管理员认证

管理员通过后台登录后访问受保护页面。

验收标准：

1. 后台登录成功后返回管理员 token。
2. 未认证访问后台时重定向或返回认证错误。
3. 管理员密码使用 BCrypt 存储。
4. 被禁用管理员不可登录。

### 需求 15：景点管理

管理员可维护景点基础数据。

验收标准：

1. 支持景点创建与编辑。
2. 支持地区、分类、经纬度、封面图、详情图录入。
3. 支持发布和下架。
4. 支持逻辑删除。
5. 支持后台筛选、搜索和分页。
6. 支持评分回刷和热度同步。

### 需求 16：攻略管理

管理员可维护攻略内容。

验收标准：

1. 支持攻略创建与编辑。
2. 支持封面图和富文本内容。
3. 支持发布和下架。
4. 支持关联景点。

### 需求 17：管理端订单查询

管理员可检索与查看订单。

验收标准：

1. 支持按状态、时间范围、订单号筛选。
2. 支持分页。
3. 支持订单详情查看。

### 需求 18：用户管理

管理员可查看用户信息并维护偏好、收藏、浏览等运营数据视图。

验收标准：

1. 列表展示昵称、头像、手机号脱敏、订单数、评分数、收藏数。
2. 用户详情展示偏好和近期订单。
3. 支持重置用户密码。
4. 后台不展示 OpenID。
5. 支持查看用户偏好、收藏与浏览视图。

### 需求 19：非功能要求

系统需满足基础安全、可靠性和可维护性要求。

验收标准：

1. 用户端和管理端均使用 JWT。
2. 业务读取默认过滤逻辑删除数据。
3. 用户端默认仅展示已发布内容。
4. 订单相关接口具备幂等保护。
5. 推荐相似度矩阵支持 Redis 存储。
6. 推荐配置、运行状态与结果缓存需可观测、可调试。

### 需求 20：仪表板

管理员可查看运营概览与趋势。

验收标准：

1. 展示总用户、总景点、总订单、总收入。
2. 展示今日新增用户、今日订单、今日收入。
3. 展示订单趋势图。
4. 展示热门景点排行。

### 需求 21：Web 用户端认证

Web 用户通过手机号和密码注册登录。

验收标准：

1. 支持手机号注册。
2. 支持手机号密码登录。
3. 手机号重复注册时返回错误。
4. 小程序绑定手机号时可合并 Web 账户。

### 需求 22：管理员管理

超级管理员可管理后台管理员账户。

验收标准：

1. 支持管理员列表查询。
2. 支持新增管理员。
3. 支持更新管理员资料。
4. 支持重置管理员密码。
5. 不允许删除自己。

### 需求 23：账户安全

用户可管理账户安全信息。

验收标准：

1. 修改密码时校验旧密码或允许首次设置密码。
2. 头像上传限制图片类型与大小。
3. 注销账户执行逻辑删除。

### 需求 24：订单超时自动取消

系统自动取消超时未支付订单。

验收标准：

1. 待支付超过 5 分钟自动取消。
2. 定时任务每分钟运行。
3. 自动写入取消时间。

### 需求 25：景点地区与分类管理

管理员维护地区树和分类树。

验收标准：

1. 支持地区按父级查询、新增、编辑、删除。
2. 支持分类按父级查询、新增、编辑、删除。
3. 分类支持图标 URL。
4. 支持按排序字段展示。

### 需求 26：推荐配置与调试

管理员可在后台查看推荐状态、调整参数并验证结果。

验收标准：

1. 支持读取和保存推荐配置。
2. 配置按 `algorithm / heat / cache` 三段组织。
3. 支持查看推荐引擎运行状态。
4. 支持手动重建相似度矩阵。
5. 支持预览指定用户的推荐结果。
6. 支持预览指定景点的相似邻居。
7. 提供推荐总览页作为调试入口。

### 需求 27：双端功能一致性

小程序端与 Web 用户端需要在核心功能覆盖、接口使用和关键用户链路上保持一致。

验收标准：

1. 推荐、发现、附近、景点、攻略、订单、互动、设置等核心页面两端均可独立运行。
2. 小程序端与 Web 用户端共用同一套核心业务接口。
3. 不允许出现仅单端可用的核心业务接口能力。
4. 允许保留登录方式等端能力差异，但需明确记录。
5. 详情返回列表后的关键状态在两端都需要尽量保持一致。

## 当前文档化结论

- 当前项目的功能缺口已经不在“四端是否具备主业务”，而在测试、性能和正式验收闭环。
- 这轮重构后，Web 用户端已经从“功能补充中”进入“与小程序端主流程基本对齐”的状态。
- 后续文档更新应继续围绕双端一致性、推荐链路验证和部署验收展开，而不是再沿用早期功能规划口径。
